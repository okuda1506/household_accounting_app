# パスワード関連仕様

## 1. 対象機能

本ドキュメントは、以下のパスワード関連機能の詳細仕様を扱う。

- ログイン済みユーザーのパスワード変更
- パスワードリセット
- 削除済みアカウントの再開

## 2. 共通パスワードポリシー

パスワード強度要件は `Password::defaults()` により共通化されている。

### 要件

- 8 文字以上
- 255 文字以下
- 大文字と小文字のアルファベットをそれぞれ 1 文字以上含む
- 数字を 1 文字以上含む

```php
Password::min(8)
    ->max(255)
    ->mixedCase()
    ->numbers();
```

## 3. ログイン済みユーザーのパスワード変更

### 3.1 エンドポイント

- `PUT /api/user/password`

### 3.2 入力項目

| 項目名 | 必須 | 説明 |
| --- | --- | --- |
| `current_password` | ○ | 現在のパスワード |
| `new_password` | ○ | 新しいパスワード |
| `new_password_confirmation` | ○ | 新しいパスワード（確認用） |

### 3.3 バリデーション設計

#### `current_password`

- 必須入力
- 入力値の正当性は Service 層で検証する
- 不一致の場合は `InvalidCurrentPasswordException` を業務例外として扱う

```php
'current_password' => ['required'],
```

#### `new_password`

- 必須入力
- 確認用項目との一致が必要
- 共通パスワードポリシーを満たす必要がある
- `bail` を指定し、整合性エラーは最初の 1 件で打ち切る

```php
'new_password' => [
    'bail',
    'required',
    'confirmed',
    Password::defaults(),
],
```

### 3.4 エラーメッセージ方針

#### `new_password`

- パスワード強度は複数条件から成るため、条件不足は複数メッセージを同時に返却する

例:

```json
[
  "新しいパスワードは8文字以上で入力してください。",
  "新しいパスワードは少なくとも大文字と小文字を1つずつ含める必要があります。",
  "新しいパスワードは少なくとも1つの数字が含まれていなければなりません。"
]
```

#### `current_password`

- 原因が 1 つに特定できるため、単一メッセージのみ返却する

### 3.5 例外処理

- 現在のパスワード不一致は `InvalidCurrentPasswordException`
- Controller 側で catch し、`422 Unprocessable Entity` を返却する

## 4. パスワードリセット

### 4.1 エンドポイント

- リンク送信: `POST /api/forgot-password`
- パスワード更新: `POST /api/reset-password`

### 4.2 入力項目

| 項目名 | 必須 | 説明 |
| --- | --- | --- |
| `token` | ○ | リセット用トークン |
| `email` | ○ | メールアドレス |
| `password` | ○ | 新しいパスワード |
| `password_confirmation` | ○ | 新しいパスワード（確認用） |

### 4.3 設計方針

- トークン検証は Laravel 標準の Password Broker に委譲する
- パスワード強度は共通ポリシーを適用する
- 有効なユーザーのみパスワードリセット対象とする

### 4.4 エラーハンドリング

- トークン不一致、期限切れ、メールアドレス不一致は Password Broker の戻り値で判定する
- パスワード強度エラーは、ログイン済み変更と同様に複数メッセージを返却する

## 5. アカウント再開

### 5.1 エンドポイント

- アカウント再開: `POST /api/reactivate-account`

### 5.2 設計方針

- 削除済みユーザーに対しては通常のパスワードリセットではなく再開フローを利用する
- `forgot-password` 実行時、対象ユーザーが削除済みであれば `ReactivateAccount` 通知を送信する
- 再開処理では Password Broker によるトークン検証後、パスワード更新と `deleted = false` を同時に行う

### 5.3 判定ルール

- 再開対象: `deleted = true`
- 通常リセット対象: `deleted = false`
- 対象状態とフローが一致しない場合は `Password::INVALID_USER` 扱いとする

## 6. 設計上の意図

- パスワード強度ルールは全機能で共通化し、運用変更時の修正箇所を減らす
- 現在パスワード不一致のような業務エラーはバリデーションエラーと分離して扱う
- 削除済みアカウントの再開をパスワード再設定フローに統合することで、UX とセキュリティの整合性を保つ
